# SNS / SQS / 死信队列（DLQ）必知知识

这份文档是通用 AWS 消息服务知识，不绑定 Token Query 项目的具体资源，但文末举例会结合项目里已有的 Alarm/Canary 场景。

## 一句话区分 SNS 和 SQS

- **SNS（Simple Notification Service）= 发布/订阅（pub/sub），推模型**。一条消息发到 Topic，Topic 会**同时推给所有订阅者**（可以是邮箱、SQS 队列、Lambda、HTTP endpoint、短信等）。发布者不知道、也不关心谁在订阅。
- **SQS（Simple Queue Service）= 队列，拉模型**。消息进队列后**只会被一个消费者取走处理一次**（同一条消息不会被两个消费者同时拿到），消费者主动轮询（poll）来取消息，不是队列推给你的。

一句话记忆：**SNS 是广播，一条消息可以有很多个接收方；SQS 是任务队列，一条消息只有一个消费者会真正处理它。**

## SNS 核心概念

- **Topic**：消息的发布点，一个 Topic 可以有多个订阅（Subscription）。
- **Subscription（订阅协议）**：常见的有 `Email`、`SQS`、`Lambda`、`HTTP(S)`、`SMS`、`Application`（推送到移动端）。
- **Fan-out 模式**：一个 Topic 同时挂多个 SQS 队列作为订阅者，实现"一个事件，多个下游系统各自独立消费"——这是 SNS+SQS 最经典的组合用法，比如一个"订单创建"事件，同时触发"发邮件""扣库存""记日志"三个互不影响的下游流程。
- **消息不会重试排队**：SNS 本身**不保证消息送达后被处理**，它只负责"推一次"。如果订阅方是 HTTP endpoint 且当时挂了，SNS 会按策略重试几次然后放弃（没有 DLQ 的话消息就丢了）——所以生产环境的 SNS 订阅通常也要配 DLQ（见下文）。

## SQS 核心概念

- **Standard Queue vs FIFO Queue**
  - Standard：吞吐量几乎无限，但**不保证顺序**、且**可能重复投递**（at-least-once，消费者必须自己实现幂等）
  - FIFO（`.fifo` 结尾）：严格顺序 + 精确一次（exactly-once，通过 `MessageDeduplicationId`），但吞吐量有限额
- **Visibility Timeout（可见性超时）**：消费者取走一条消息后，这条消息在超时时间内对其他消费者"不可见"，防止被重复处理。如果消费者在超时时间内没有主动删除（ack）这条消息，超时后消息会**重新变为可见**，被再次投递——这是 SQS 实现"重试"的核心机制。
  - 常见坑：Lambda 处理耗时 > Visibility Timeout，会导致同一条消息被多个并发 Lambda 实例同时处理。**Visibility Timeout 必须设置得比消费者最长处理时间还长**（AWS 官方建议至少 6 倍 Lambda 超时时间）。
- **长轮询（Long Polling）**：`ReceiveMessage` 时设置 `WaitTimeSeconds > 0`，SQS 会在没有消息时等待一段时间再返回空，而不是立刻返回空结果——省请求次数、省钱，生产环境基本都应该开长轮询而不是短轮询。
- **Redrive Policy（重新投递策略）**：定义"一条消息最多被取出多少次（`maxReceiveCount`）还没被成功删除，就转发到死信队列"，这是连接普通队列和 DLQ 的桥梁。

## 死信队列（Dead Letter Queue, DLQ）到底是什么

DLQ **本质上就是一个普通的 SQS 队列**，没有任何特殊的 AWS 资源类型——特殊的地方在于"谁把消息转发到这里"这条**配置**（Redrive Policy），而不是队列本身。

### 工作机制

1. 给业务队列（或 SNS 订阅）配置 Redrive Policy，指定：
   - 目标 DLQ 的 ARN
   - `maxReceiveCount`：消息被取出但一直没被成功删除（ack）的次数上限
2. 消息每次被消费者取走但没在 Visibility Timeout 内删除，`ReceiveCount` 就 +1
3. 一旦 `ReceiveCount` 超过 `maxReceiveCount`，SQS 自动把这条消息**移动**（不是复制）到 DLQ，业务队列里这条消息就没了
4. 消息进了 DLQ 之后**不会自动重试**，也不会自动过期清理业务逻辑——它就静静地待在 DLQ 里，等你手动或写脚本去处理

### 为什么需要 DLQ

- **防止"毒消息"（poison message）卡死队列**：如果某条消息本身有问题（格式错、触发代码 bug），消费者每次处理都失败又不删除，这条消息会不断重新出现、反复消费失败，占用消费者资源、阻塞后面正常消息的处理。DLQ 把这种消息隔离出来，业务队列能继续往前走。
- **保留失败现场，方便排查**：DLQ 里的消息本身就是"失败样本"，可以直接查看内容、日志关联、复现问题，而不是消息丢了之后两眼一抹黑。
- **可以做二次处理/告警**：给 DLQ 配一个 CloudWatch Alarm（`ApproximateNumberOfMessagesVisible > 0`），一旦有消息掉进 DLQ 就立刻通知人工介入，这是最常见的用法。

### SNS 也能配 DLQ

SNS 的每个 Subscription 也可以单独配一个 DLQ——如果 SNS 尝试推送给某个订阅方（比如一个 Lambda 或 HTTP endpoint）连续失败超过重试次数，这条消息会被送进这个订阅专属的 DLQ，而不是直接丢弃。**注意 SNS 的 DLQ 是挂在"订阅"级别，不是"Topic"级别**——同一个 Topic 下不同订阅可以配不同的 DLQ，或者只给某几个订阅配。

## 常见误区 / 容易踩的坑

1. **以为 DLQ 会自动重试消息** —— 不会。DLQ 只是"存放失败消息的地方"，重新处理需要你自己写脚本 `ReceiveMessage` → 处理 → 删除，或者用 AWS 控制台的 "Start DLQ redrive" 功能手动把消息送回原队列重试。
2. **Visibility Timeout 设置过短** —— 消费者还没处理完消息就"可见"了，导致同一条消息被多个消费者同时处理，业务逻辑不幂等的话会出重复副作用（比如重复扣款、重复发邮件）。
3. **FIFO 队列的 DLQ 也必须是 FIFO 队列** —— 类型要匹配，Standard 队列的 DLQ 必须也是 Standard。
4. **只给业务队列配 DLQ，却没给 DLQ 本身配监控** —— DLQ 是"最后一道防线"，如果没人盯着它，消息进去了但没人发现，等于没配。至少要配一个基于 `ApproximateNumberOfMessagesVisible` 的 Alarm。
5. **`maxReceiveCount` 设置为 1** —— 意味着消息第一次处理失败就直接进 DLQ，完全没有重试的容错空间，通常建议至少 3-5 次再进 DLQ（除非你确定失败大概率是消息本身有问题，重试也没用）。
6. **SNS 消息本身没有重试保底** —— 如果订阅方是普通 HTTP endpoint 又没配 DLQ，失败重试次数用完后消息直接丢弃，什么记录都不会留下。生产环境的 SNS 订阅（尤其是 Lambda/HTTP）建议都配上 DLQ。

## 沙盘推演：任务分发该用"一个队列 + type 字段"还是"多个队列"

场景：一个 boss 要给团队派发任务——`[前端开发1, 前端开发2, 后端开发1, 后端开发2]`，消费者是前端张三、李四，后端王五、赵六。队列怎么设计更合理？

**方案 A：一个队列，消息里带 `type` 字段区分前端/后端**——不推荐，这是一个常见反模式。

原因：**SQS 没有"按内容选择性拉取"的能力**。张三 `ReceiveMessage` 的时候，队列只会给他"排在最前面的任意一条消息"，不管这条消息的 `type` 是不是 `frontend`。如果张三拉到一条 `type: backend` 的任务，他只能选择不删除、等它超时复活，交给别人重新拉——这一来一回会白白消耗一次 `ReceiveCount`。如果一条消息运气不好被好几个"类型不对"的消费者反复摸到，`ReceiveCount` 很容易在没有任何真实处理失败的情况下就超过 `maxReceiveCount`，被误判成"失败"扔进死信队列——**这是纯粹由路由错误导致的假死信，不是任务本身有问题**。

**方案 B：拆成两个队列（`frontend-queue` / `backend-queue`）**——更合理。

- 张三/李四只 poll `frontend-queue`，王五/赵六只 poll `backend-queue`，路由 100% 准确，不会有"拉错类型又扔回去"的浪费
- 每个队列可以有**独立的 Visibility Timeout**（前端任务和后端任务耗时预期可能完全不同）
- 每个队列可以配**独立的 DLQ / `maxReceiveCount` / CloudWatch Alarm**——出问题时立刻知道是前端任务堆积了还是后端任务挂了，不用混在一起排查（这跟本项目把 Lambda canary 和 Go canary 拆成两个、各自独立告警是同一个设计取舍，见 [todayToDo.md](../todayToDo.md) Part 2）

**更进一步的做法：SNS 做路由分发，Boss 不需要知道下游有几个队列**

如果 Boss（生产者）要跟着团队数量变化去决定"发到哪个队列"，团队一多就很难维护。更好的做法是在两个队列前面加一层 SNS Topic：Boss 只管往一个 Topic 发任务（带上 `team: frontend` / `team: backend` 这个 Message Attribute），`frontend-queue` 和 `backend-queue` 分别以**订阅过滤策略（Filter Policy）**订阅这个 Topic，只接收匹配自己 `team` 属性的消息——这是本文前面提到的 Fan-out 模式的进阶用法。以后哪怕再加一个"移动端开发队列"，Boss 那边完全不用改代码，只要新加一个订阅+过滤条件就行。

**结论**：多队列（方案 B），理想情况下再加一层 SNS + Filter Policy 做路由，让生产者跟"下游到底有几种消费者"解耦。

## 拆队列的判断标准：不是看"类型标签"，是看"消费者能力边界"

上面这个沙盘推演可以再抽象出一条更通用的原则：

**一个 SQS 队列的"职责边界" = "订阅它的消费者是不是同一批、能力对等、可以互换处理这个队列里任意一条消息"。**

判断要不要拆队列，看的不是"消息类型标签是不是不一样"，而是看：**这个队列里的消息，是不是所有订阅它的消费者都有能力处理、而且处理方式/超时/重试策略都差不多**。

- 消费者能力不对等（比如张三不会处理后端任务）→ 必须拆，不然会出现前面说的"拉错任务又扔回去"导致的假死信问题
- 就算消费者能力对等，如果不同任务类型该有不同的运维参数（超时时间、重试次数、DLQ 策略），拆开也能让这些参数各自独立调，不用被迫取一个折中值
- 拆开之后每个队列能有独立的 DLQ / Alarm，出问题时直接定位到是哪一类任务出的问题

**这不是"消息里只要有 type 字段就必须拆队列"的机械规则**。如果一个队列的消费者是同一批人/同一套处理逻辑，只是消息内容有细分（比如都是"前端任务"，细分成"构建任务"和"部署任务"，两种都归同一批前端消费者处理，超时/重试策略也一样），完全可以放在同一个队列里——这时候 `type` 字段只是告诉消费者"具体该跑哪段处理逻辑"，不是用来做"排除性路由"，不存在"消费者拿到不该拿的消息"这个问题，也就没有假死信的风险。

一句话总结这条边界：**拆队列拆的是"消费者能力/处理策略不同"这条线，不是"消息标签不同"这条线**——只是现实中大部分"多种任务类型"场景，这两条线恰好是重合的（不同类型任务通常对应不同的消费者能力或不同的处理需求），所以才会得出"多种任务类型该拆多个队列"这个直觉上正确的结论。
