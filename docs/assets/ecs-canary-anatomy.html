Cluster / Service / Task 与 ALB / Target Group 到底是什么关系

两棵树，一座桥。左边是 ECS 管"要跑什么"，右边是 ALB 管"流量给谁"，中间靠 Target = Task 的 ENI IP 连起来。往下先拆开"桥"具体是怎么搭的，再看 learning-go-canary-service 这次真实 v1→v2 灰度发布时，两边怎么互相触发。

01

静态结构 — 谁装着谁

左边是"计算"层级，右边是"流量"层级，两边平时互不知道对方存在——是 ECS 的 deployment controller 在部署时把两边接起来的。

计算侧 · ECS
Cluster learning-go-canary-cluster 作用 一堆 Service 的资源边界，本身不跑代码
Service learning-go-canary-service strategy CANARY　desired 2
职责 维持任务数量 + 按 canary 节奏推进发布
Task Definition family learning-go-canary :1 -text=v1　:2 -text=v2
Task 10.0.3.30:5678 自己的 ENI，独立 IP
Task 10.0.4.208:5678 自己的 ENI，独立 IP
Target = Task 的 ENI IP
流量侧 · ALB
Load Balancer learning-go-canary-alb scheme internal　作用 接请求，不关心谁在跑
Listener HTTP : 80

Target Group learning-tg-a
Target Group learning-tg-b
关键 ALB / Target Group 的配置你自己动手建，但发布期间的 Listener 权重和 Target 注册，是 ECS 的 deployment controller 全权接管的 —— 你在 Update service 里选了新 revision 之后，接下来 Rule 权重从 100/0 走到 10/90 再到 0/100，全是 ECS 自己在改，不需要（也不应该）手动去 ALB 那边调。

Service 本身只是一份声明式配置（写着"该管哪条 Rule、用哪个角色"），它自己不会去改 ALB——真正动手的是 ECS 控制面里的后台逻辑，借着你给的 IAM 角色去执行：

1 你 触发 update-service

控制台点 Update，或者 CLI 调用——只是告诉 ECS "该切到哪个 Task Definition revision 了"

→
2 Service（声明式配置） 只是记录该管哪条 Rule、用哪个角色

Service 定义里的 productionListenerRule、testListenerRule、roleArn 都只是配置字段，Service 这个资源本身不执行任何动作

→
3 ECS deployment controller 后台按 canary 节奏决策

AWS 后台运行的控制逻辑，按你配的 canary percent / bake time 时间表，决定"现在该把权重改成多少"

→
4 借用 roleArn 调用 elbv2:ModifyRule

用 Service 里配的那个角色（learning-go-canary-esc-role）的身份去调 ELB API，真正改写 ALB Rule 里两个 Target Group 的转发权重——不是用你自己的身份，也不是 Service 资源自己"跑"起来去做

02

"注册进 Target Group" 具体是谁做了什么

这四步是任务从"刚启动"到"能真正接住流量"必须经过的关联过程——注意"注册"这个动作既不是任务自己发起的，也不是你手动去 ALB 控制台点的。

1 Fargate 任务启动，分配 ENI

awsvpc 网络模式下，每个任务开机时会拿到一张独立网卡和私有 IP，比如 10.0.4.208。这一步任务只是"活了"，跟 ALB 还没有任何关系

→
2 ECS 控制面 读 Service 的 loadBalancers 配置

Service 定义里写死了 containerName: http-echo、containerPort: 5678 该挂到哪个 Target Group——发布期间新任务固定挂到 alternateTargetGroupArn（这次是 tg-b）

→
3 ECS → ELB API 调用 RegisterTargets

ECS 代你调用 elbv2:RegisterTargets(target-group, ip:port)——这就是事件日志里 "registered 1 targets" 那条的真实来源，全程没人手动点过 ALB 控制台

→
4 ALB 健康检查，转正

Target Group 按自己配置的健康检查（path /，端口 5678）反复探测这个 IP，状态从 initial 变成 healthy 后，Listener Rule 才会真的把流量转发过去

why tg-b 为什么这次新任务注册进的是 tg-b 不是 tg-a？——当前生产流量在哪个 Target Group（这次是 tg-a），ECS 就把新版本任务塞进另一个当"候选槽位"（这次是 tg-b，即 alternateTargetGroupArn）。等这次发布彻底切完，两个槽位的角色会对调：下一次发布，新任务反过来会注册进 tg-a。任务停止时是同样的反向过程——ECS 调用 DeregisterTargets，ALB 再给一段 connection draining 时间（Target Group 的 Deregistration delay，你建的时候默认是 300 秒）才彻底摘掉。
03

一个请求进来，怎么被分到具体某个 Task

分两层决策，橙色这一层管"灰度比例"，灰色这一层跟版本无关，只是普通负载均衡。

第一层 · Listener Rule 每个请求进来，按 Rule 里配的权重（tg-a 90% / tg-b 10%）做一次加权抽签，决定这次请求丢给哪个 Target Group。灰度百分比只发生在这一层——量大了之后请求数趋近这个比例，单次请求不是精确切分的。
第二层 · Target Group 内部 确定丢给哪个 Target Group 后，ALB 只在这个组里健康的 Task 之间做普通负载均衡（默认轮询），挑一个转发过去。这一层不区分版本，纯粹是"组内谁健康就转给谁"。
